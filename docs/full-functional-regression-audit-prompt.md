# Промпт для полного функционального и регрессионного аудита BonkScanner

Работаем в репозитории `F:\Python\MegabonkReroll`.

Нужен полный регрессионный и функциональный аудит BonkScanner — последовательно,
вкладка за вкладкой, включая каждый элемент, карточку, кнопку, настройку,
счётчик, виджет и связанный фоновый процесс.

## Главная цель

Найти функциональные регрессии и архитектурные ошибки, похожие на уже
обнаруженные проблемы:

1. Случайное чтение мусора из памяти дало предметам значения вроде
   `759271589`, после чего:
   - испортился Live Stats;
   - Stage Summary накопил огромные значения;
   - повреждённые значения могли попасть в записи;
   - один невалидный snapshot отравил дальнейшее состояние.
2. При переходе на Stage 3 временные значения Chests/Pots были ошибочно приняты
   за Stage 4:
   - временное состояние во время загрузки стало постоянным;
   - tracker монотонно перешёл на неверную стадию;
   - Stage Summary больше не мог самостоятельно восстановиться.

Такие ошибки раньше не проявлялись и в нормальной архитектуре не должны были
быть возможны. Нужно проверить, не появились ли аналогичные проблемы в
остальных функциях после рефакторинга обмена данными, refresh coordinator,
LiveRunTracker и разделения источников данных.

## Основной принцип аудита

Для каждой функции прослеживай полный путь данных:

```text
источник данных
→ чтение памяти/API/config
→ валидация
→ snapshot/runtime state
→ tracker/cache
→ расчёты
→ UI
→ запись
→ replay
→ compare
→ overlay
→ Twitch
```

Недостаточно проверить, что функция или виджет существуют. Нужно доказать, что
они:

- получают правильные данные;
- получают их из правильного источника;
- корректно переживают временные ошибки;
- не принимают переходное состояние за окончательное;
- не отравляют накопительное состояние;
- правильно сбрасываются между забегами и стадиями;
- одинаково работают во всех потребителях данных.

### Шаг 5. Проверка опасных сценариев

Каждую функцию проверяй хотя бы концептуально, а где возможно — существующими
тестами или read-only воспроизведением.

Обязательные сценарии:

- запуск приложения без игры;
- подключение игры после запуска приложения;
- запуск приложения посреди забега;
- подключение на Stage 2, Stage 3 и Stage 4;
- обычный переход между Stage 1–4;
- смена map seed и stage pointer;
- короткое исчезновение pointer;
- временный ноль;
- временное пустое значение;
- частично прочитанный dictionary/list;
- повторное использование старого memory object;
- один ошибочный snapshot между двумя правильными;
- несколько последовательных ошибок;
- восстановление после ошибки;
- очень большое, отрицательное, NaN или бесконечное значение;
- рассинхрон двух последовательных memory reads;
- duplicate entries;
- рост и уменьшение stack/count;
- пауза игры;
- окончание забега;
- быстрый reset;
- новый забег с тем же или другим seed;
- UI-вкладка открыта;
- UI-вкладка закрыта, но данные нужны overlay/Twitch/recording;
- recording включён и выключен;
- snapshot interval больше частоты live refresh;
- загрузка старой записи;
- повреждённая или частичная запись;
- смена настроек во время работы;
- закрытие приложения во время активного worker/timer.

### Шаг 6. Поиск архитектурных классов ошибок

Ищи не только конкретные неправильные числа, но и системные причины:

- сырое значение принимается без проверки;
- pointer или ID интерпретируется как count;
- transient state принимается за постоянное;
- временный ноль используется как reset;
- пустой результат неотличим от ошибки чтения;
- partial snapshot помечается как полностью доступный;
- последний корректный кеш заменяется повреждённым;
- накопительный tracker нельзя откатить после ошибочного скачка;
- stage/run state может двигаться только вперёд и не умеет самовосстанавливаться;
- разные refresh-задачи обновляют зависимые поля несинхронно;
- UI и recording получают snapshot из разных моментов времени;
- recorder сохраняет данные, которые UI уже признал невалидными;
- replay повторно интерпретирует уже агрегированные значения;
- old/new recording schema несовместимы;
- значения сбрасываются при переключении вкладки;
- функция обновляется только когда вкладка видима, хотя нужна overlay/Twitch;
- callback перестал подключаться после рефакторинга;
- timer после исключения больше не планирует следующий tick;
- два worker одновременно изменяют один state без блокировки;
- lock защищает контейнер, но не логическую транзакцию;
- config нормализуется иначе при загрузке и сохранении;
- fallback считается успешным при частичном чтении;
- тесты проверяют только happy path и используют полностью стабильный FakeMemory.

### Шаг 7. Проверка тестов

Для каждой функции найди существующие тесты и ответь:

- что именно они доказывают;
- какие реальные сценарии они не покрывают;
- не закрепляют ли они ошибочное поведение;
- соответствуют ли FakeMemory/FakeTracker реальному lifecycle;
- проверяют ли они transient/partial/stale состояния;
- проверяют ли они взаимодействие нескольких snapshots;
- проверяют ли они reset и восстановление;
- проверяют ли они одинаковый результат в Live Stats, recording и replay.

На аналитической фазе не добавляй намеренно падающие тесты в рабочее дерево. Для
проверки гипотез можно использовать временные read-only скрипты или существующие
тесты.

После моего разрешения на исправление:

1. Сначала добавь регрессионный тест, воспроизводящий ошибку.
2. Исправь первопричину, а не только отображение.
3. Проверь все потребители общего значения.
4. Запусти focused tests.
5. Затем запусти полный `pytest src/tests`.
6. Покажи точные изменённые файлы.
7. Не коммить без отдельной просьбы.

## Формат отчёта по каждой вкладке

1. Название вкладки и границы аудита.
2. Полный список проверенных элементов.
3. Использованная документация.
4. Ожидаемое поведение.
5. Фактическая архитектура и data flow.
6. Отличия от `main`.
7. Найденные проблемы.

Для каждой проблемы укажи:

- Severity: Critical / High / Medium / Low;
- подтверждённая или вероятная;
- пользовательский эффект;
- точный сценарий воспроизведения;
- первопричина;
- файлы и строки;
- почему существующие тесты это пропустили;
- какие другие вкладки/интеграции затронуты;
- какой regression test нужен;
- предлагаемое направление исправления без реализации на аналитической фазе.

Затем укажи:

8. Потенциально опасные места, которые пока не удалось доказать.

10. Проверенные части без найденных проблем.
11. Запущенные тесты и результаты.
12. Рекомендованный порядок исправлений.
13. Следующую вкладку по плану — но не начинай её без моей команды.

## Критерий качества

Фраза «тесты проходят» не является доказательством отсутствия бага.

Аудит считается завершённым только тогда, когда для каждого элемента вкладки
понятны:

- источник истины;
- документированное назначение;
- жизненный цикл;
- правила валидации;
- правила кеширования;
- reset/recovery;
- все потребители;
- отличие от `main`;
- имеющееся тестовое покрытие;
- риски transient и partial данных.
